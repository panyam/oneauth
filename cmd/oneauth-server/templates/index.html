OneAuth Demo

Federated authentication for distributed services

What this demo shows

Multiple independent Apps register with this central auth server, authenticate their own users, and mint resource-scoped JWTs. Resource Servers validate those tokens using a shared signing key database — without ever talking to the apps.

This is the same pattern used by Auth0, Okta, and Firebase for federated identity — but here the trust anchor is a shared PostgreSQL signing_keys table instead of a JWKS endpoint.

Try it — suggested order:

  1. Sign up here on the Auth Server (the Login card below) to see the dashboard and registered apps.
  2. Open DrawApp (:3001) — sign up with a different email. This is a completely separate user database.
  3. Click "Get Resource Token" in DrawApp to mint a JWT signed with DrawApp's secret.
  4. Click "Validate" to send that JWT to a Resource Server. It has never seen DrawApp — it trusts the shared KeyStore.
  5. Repeat with ChatApp (:3002). Try validating tokens on both Resource Server A and B.

Architecture (what's running)

PostgreSQL :5432  ← shared signing_keys table (trust anchor)
  ├─ Auth Server :9999       writes keys at app registration
  ├─ Resource Server A :4001 reads keys at token validation
  ├─ Resource Server B :4002 reads keys at token validation
  ├─ DrawApp :3001           registered app, owns users, mints JWTs
  └─ ChatApp :3002           registered app, owns users, mints JWTs
